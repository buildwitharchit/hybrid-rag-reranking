The Core Problem This Solves
Naive RAG has one fatal flaw: it retrieves by semantic similarity alone. If a user asks "What are the side effects of metformin?", naive RAG embeds that query and finds chunks that sound similar — but might miss the exact document that uses clinical language like "adverse reactions" or "contraindications." You get semantically close but factually wrong results.
This project fixes that with four layers of increasingly expensive-but-accurate retrieval.

Part 1 — Document Ingestion
Before any query happens, you need to get documents into the system.
What happens:

User uploads a PDF, pastes a URL, or drops in plain text
The document gets chunked — split into overlapping pieces (e.g., 512 tokens with 50-token overlap)
Overlap matters because a fact might sit at a chunk boundary — without overlap you'd lose it
Each chunk gets two representations stored simultaneously:

A dense vector (float array) via sentence-transformers/all-MiniLM-L6-v2 → stored in ChromaDB
Added to a BM25 index (just the raw text, no vectors) → stored in memory via rank_bm25



So every chunk lives in two places. This is the foundation of hybrid search.

Part 2 — Query Expansion
When a user types a query, you don't search immediately. First you expand it.
What happens:
You send the original query to the LLM with a prompt like:

"Generate 3 alternative phrasings of this question that preserve meaning but use different vocabulary."

Example:

Original: "How does transformer attention work?"
Variant 1: "What is the self-attention mechanism in neural networks?"
Variant 2: "Explain the key-query-value computation in transformers"
Variant 3: "How do transformers decide what to focus on?"

Why this helps: Different phrasings hit different chunks. Someone who wrote documentation might use "self-attention" while another used "attention mechanism." You cast a wider net without changing what the user actually wanted.
Now you have 4 queries (original + 3 variants). Each one goes into both retrieval systems.

Part 3 — Hybrid Retrieval (Dense + Sparse)
This runs in parallel for all 4 query variants.
Dense Search (ChromaDB + sentence-transformers)

The query gets embedded into a vector
ChromaDB does a cosine similarity search and returns top-20 chunks
Good at: paraphrases, synonyms, conceptual similarity
Bad at: exact keyword matches, rare proper nouns, numbers

Sparse Search (BM25)

BM25 is essentially a smarter TF-IDF — it scores documents by how often query terms appear, weighted by how rare those terms are across the corpus
No embeddings, pure term frequency math
Good at: exact keyword matches, technical terms, IDs, names
Bad at: understanding synonyms or paraphrases

These two search methods are complementary — one's weakness is the other's strength.

Part 4 — Reciprocal Rank Fusion (RRF)
You now have up to 8 ranked lists (4 queries × 2 retrieval methods), each with up to 20 results. You need to merge them into one list.
RRF formula for each chunk:
RRF_score(chunk) = Σ  1 / (k + rank_in_list)
where k is typically 60 (a smoothing constant).
Why this works: A chunk that appears at rank 1 in one list and rank 3 in another gets a high combined score. A chunk that appears at rank 1 in many lists gets an even higher score. Rank position matters more than the raw similarity score — this makes the fusion robust to score scale differences between dense and sparse systems.
After RRF, you have one merged ranked list of ~20-40 unique chunks.

Part 5 — Cross-Encoder Re-ranking
This is the most expensive step and the biggest quality jump.
The difference between bi-encoders and cross-encoders:
Bi-encoder (what you used for dense search)Cross-encoder (re-ranker)How it worksEncodes query and chunk separately, compares vectorsEncodes query + chunk together, outputs a relevance scoreSpeedFast (pre-compute chunk vectors)Slow (must run for every query-chunk pair)AccuracyGoodMuch better — sees interactions between query and chunk
What happens:

You take the top-20 chunks from RRF
For each chunk, you feed [query, chunk_text] as a pair into cross-encoder/ms-marco-MiniLM-L-6-v2
It outputs a single relevance score (not a vector)
You re-sort by this score and take top-5

The cross-encoder catches things like: the chunk mentions the query topic but in a contradicting or unrelated context — the bi-encoder would miss this, the cross-encoder catches it.
If using Cohere Rerank API, you outsource this step — same idea, just a hosted model.

Part 6 — Context Assembly + LLM Generation
Top-5 chunks are assembled into a context block with source metadata (document name, page, chunk ID). This gets sent to the LLM via OpenRouter with a prompt like:
Answer the question using only the provided context.
Cite which source each claim comes from.

Context:
[Chunk 1 — source: doc_name.pdf, page 3] ...
[Chunk 2 — source: url.com, para 7] ...

Question: {user_query}
The LLM generates an answer that cites [Source 1], [Source 2], etc. — so the user can see exactly which chunk each claim came from.

Part 7 — The Eval System (This is what makes the project stand out)
This is how you get real numbers to put in the README.
You build a ground truth eval set — about 50-100 query/answer/relevant-doc triples. Example:
json{
  "query": "What is the capital of France?",
  "ground_truth_answer": "Paris",
  "relevant_chunk_ids": ["chunk_042", "chunk_107"]
}
Then you run both naive RAG and your hybrid system on every query and compute:
NDCG@5 (Normalized Discounted Cumulative Gain)
Measures whether relevant docs appear near the top of retrieved results. A relevant doc at rank 1 scores more than at rank 4. Normalized so max = 1.0. You compare naive vs hybrid.
MRR (Mean Reciprocal Rank)
For each query: what rank was the first relevant chunk? Score = 1/rank. Average across all queries. Tells you "does the right doc come up first?"
Faithfulness (RAGAS)
RAGAS sends the generated answer + source context to an LLM judge that checks: "Does every claim in the answer actually come from the retrieved context?" Score 0-1. Catches hallucinations where the LLM ignored the context.
Latency
You time both pipelines. Hybrid is obviously slower (query expansion + re-ranking adds ~1-2 seconds). The latency/quality tradeoff curve shows this honestly.

PART 1 — CHUNKING STRATEGY
What is chunking and why does it exist?
LLMs have a context window limit. You cannot feed a 200-page PDF into a prompt. Even if you could, attention degrades over long contexts — the model pays less attention to things buried in the middle. So you break documents into smaller pieces and only retrieve the relevant pieces.
But here's the real problem: chunking is a lossy operation. Every time you split a document, you risk cutting a fact in half. A sentence that starts on page 3 and ends on page 4 might get split across two chunks. Neither chunk alone contains the complete fact.
This is why chunking strategy is not trivial — it directly determines your retrieval ceiling. Bad chunking = no amount of fancy retrieval fixes it.

The different chunking strategies
Fixed-size chunking (what most people start with)
Split every N tokens, regardless of sentence/paragraph boundaries.
chunk_size = 512 tokens
overlap = 50 tokens
The overlap means chunk 2 starts 50 tokens before chunk 1 ends. This handles the boundary problem partially.

✅ Simple, predictable, fast
❌ Cuts sentences mid-thought, no semantic awareness
❌ A 512-token chunk might have 3 unrelated topics crammed together

Sentence-boundary chunking
Use a sentence tokenizer (spaCy, NLTK) to only cut at sentence ends. Group sentences until you hit the token limit.

✅ Preserves grammatical units
❌ Sentence lengths vary wildly — some chunks are 50 tokens, some are 400
❌ A paragraph might span 20 short sentences and one very long thought

Recursive character splitting (LangChain's approach)
Try to split on paragraphs first (\n\n), then sentences, then words, then characters — in order of preference. Only go to the next level if the chunk is still too big.

✅ Respects document structure when possible
✅ Falls back gracefully
✅ This is the most practical general-purpose strategy

Semantic chunking
Embed every sentence. When the cosine similarity between consecutive sentences drops below a threshold, that's a topic boundary — cut there.

✅ Chunks actually correspond to coherent topics
❌ Expensive — you're embedding every single sentence during ingestion
❌ Threshold tuning is tricky
❌ Overkill for most use cases

Document-structure-aware chunking
For PDFs: respect headings, sections, bullet points. For code: respect function/class boundaries. For HTML: respect <section>, <article> tags.

✅ Best quality if your documents have consistent structure
❌ Brittle — real-world PDFs are messy, headings don't always parse cleanly


Why this project uses fixed-size with overlap + lets user choose
You expose chunking strategy as a sidebar selector because there is no universally best strategy. Legal documents → sentence boundary. Code → function-level. Wikipedia-style articles → recursive. Giving the user control and showing how it affects retrieval quality is itself a learning.
The overlap of 50-100 tokens is non-negotiable. It's cheap insurance against boundary cuts.

What token size to choose
Chunk sizeGood forProblem128-256 tokensPrecise, specific factsLoses context — a fact without surrounding context is ambiguous512 tokensGeneral purpose sweet spotSome topic mixing1024+ tokensLong-form reasoningDense chunks hurt precision — many irrelevant sentences
The all-MiniLM-L6-v2 model has a 512-token maximum input. Anything longer gets truncated silently. So your chunk size should never exceed 512 tokens for this embedding model. This is a constraint that shapes everything downstream.

PART 2 — EMBEDDINGS & DENSE SEARCH
What embeddings actually are
An embedding model converts text into a vector — a list of ~384 floating point numbers (for MiniLM). These numbers encode semantic meaning in a geometric space. Sentences with similar meanings have vectors that point in similar directions (high cosine similarity).
"The cat sat on the mat"  →  [0.23, -0.87, 0.12, ...]  (384 numbers)
"A feline rested on a rug" →  [0.21, -0.84, 0.15, ...]  (similar direction)
"Stock markets rallied"    →  [-0.45, 0.33, -0.67, ...]  (very different)
The model was trained via contrastive learning — pairs of semantically similar sentences were pulled together in vector space, dissimilar ones pushed apart. It learned what "similar meaning" means across millions of examples.

Why all-MiniLM-L6-v2 specifically
sentence-transformers library has dozens of models. Here's how they compare:

Vector databases store embeddings and let you query by similarity. Here's the landscape:

ChromaDB runs entirely in-process — no server to spin up, no Docker, just pip install chromadb. For a portfolio project this is exactly right. The code stays simple. Anyone can clone and run it.
The weakness of ChromaDB: it uses HNSW (Hierarchical Navigable Small World) for approximate nearest neighbor search. HNSW is excellent but at 10M+ vectors it gets slow. For your use case (thousands to tens of thousands of chunks), it's perfect.

What HNSW actually does (so you can explain it in interviews)
Brute-force vector search = compare your query vector against every stored vector. At 100K vectors × 384 dimensions, that's 38.4M multiplications per query. Slow.
HNSW builds a multi-layer graph. Top layers have few nodes with long-range connections. Bottom layers are dense with local connections. Search starts at the top, navigates toward similar nodes, drops to lower layers for refinement. It finds approximate nearest neighbors in O(log n) time instead of O(n).
The tradeoff: it's approximate — it might miss the true nearest neighbor 1-2% of the time. For RAG this is acceptable.

PART 3 — SPARSE SEARCH (BM25)
The fundamental limitation of dense search
Dense search understands meaning but is bad at exact term matching. Consider:

Query: "What is the error code E-4302 in the manual?"
Dense search finds chunks about error handling, error codes generally, manual sections about errors
BM25 finds the exact chunk containing "E-4302" because it looks for term frequency

For technical documentation, medical terms, legal citations, product codes, names — BM25 often wins. Dense search has no concept of "this specific string matters."

How BM25 actually works
BM25 is a probabilistic retrieval model. The score for a document D given query Q is:
BM25(D, Q) = Σ IDF(qi) × [f(qi, D) × (k1 + 1)] / [f(qi, D) + k1 × (1 - b + b × |D|/avgdl)]
Breaking this down:

f(qi, D) = how many times query term qi appears in document D (term frequency)
IDF(qi) = inverse document frequency — rare terms score higher than common terms. "The" appears everywhere so it has near-zero IDF. "Metformin" appears rarely so it has high IDF
|D|/avgdl = document length normalization — a long document mentioning a term once shouldn't beat a short document mentioning it once
k1 (usually 1.2-2.0) = term frequency saturation — prevents a term appearing 100 times from scoring 100× better than appearing 5 times
b (usually 0.75) = length normalization strength

BM25 requires no training, no GPU, no API. It's pure statistics on term frequencies.
Alternatives to BM25:
MethodWhat it isvs BM25TF-IDFOlder predecessorBM25 strictly better — added length normalization and saturationSPLADELearned sparse embeddingsBetter quality, needs GPU/model, not truly "sparse"BM25+BM25 variantFixes some edge cases with very low term frequenciesElasticsearchBM25 under the hoodSame algorithm, just distributed and production-grade
rank_bm25 library is a pure Python BM25 implementation. Dead simple. You tokenize chunks, build an index, query it. For a portfolio project this is perfect. Production system? Elasticsearch would be more appropriate.

The information retrieval insight
Why does hybrid work? It comes from a paper observation: dense and sparse retrievers have low overlap in their failure modes. When dense fails, sparse often succeeds, and vice versa. Studies on BEIR benchmark consistently show hybrid outperforms either alone by 5-15% NDCG. This is the empirical justification for the whole approach.

PART 4 — QUERY EXPANSION
Why queries fail
Users write short, ambiguous queries. "Python memory issue" could mean:

Memory leak in Python code
Python's garbage collection behavior
RAM usage of Python processes
Python's memory model (references, etc.)

A single embedding of this query captures one semantic direction. Expansion generates phrasings that cover multiple directions.

How you implement it
You prompt the LLM:
You are a search query optimizer. Given a user's question, generate 3 
alternative phrasings that:
- Preserve the original intent
- Use different vocabulary
- Approach the topic from different angles

Return as a JSON array of strings.

Original question: {query}
The LLM returns:
json["Python memory leak detection", 
 "how to fix high RAM usage in Python program", 
 "Python garbage collection and memory management"]
You now search all 4 queries (original + 3 variants) and fuse results.

HyDE is particularly interesting and worth knowing about: instead of rephrasing the question, you ask the LLM "Write a paragraph that would answer this question" — then embed that hypothetical answer. The intuition is that a good answer is closer in embedding space to relevant chunks than the question itself. Works surprisingly well.

The cost/latency math
Query expansion adds one LLM call before retrieval. With OpenRouter at ~$0.001 per query, that's negligible. Latency-wise, it's 500ms-1s. For a chatbot this is acceptable. For a high-throughput system (1000 QPS) it becomes a bottleneck — you'd cache expansion results or use a smaller local model.

PART 5 — RECIPROCAL RANK FUSION
The merge problem
After running 4 queries × 2 retrieval methods = 8 ranked lists, each with up to 20 results, you have a messy collection of overlapping chunks with inconsistent scores.
Can you just average the scores? No. BM25 scores and cosine similarity scores are on completely different scales. BM25 might return scores of 2.3, 1.8, 1.2. Cosine similarity returns 0.87, 0.82, 0.79. A direct average would be dominated by BM25's scale.
Could you normalize them? You could min-max normalize each list, then average. This is called linear combination fusion and it works but has a problem: it's sensitive to outliers. If BM25 returns one super high-scoring document and the rest are low, normalization distorts everything.

Why RRF is elegant
RRF doesn't use scores at all. Only ranks matter.
RRF_score(chunk) = Σ  1 / (60 + rank_in_list_i)
For each of your 8 lists, if a chunk appears at rank r, it contributes 1/(60+r) to its total score.
Why 60? It's a smoothing constant that prevents rank 1 from being overwhelmingly dominant. Empirically, 60 was found to work well in the original RRF paper (Cormack et al., 2009). Some implementations use 0 or smaller values — this gives more weight to top ranks.
Example:
Chunk A:

Dense search on query 1: rank 3 → contributes 1/63 = 0.0159
Sparse search on query 1: rank 1 → contributes 1/61 = 0.0164
Dense search on query 2: rank 5 → contributes 1/65 = 0.0154
Doesn't appear in other lists

Total = 0.0477
Chunk B:

Appears at rank 12 in all 8 lists
Each contributes 1/72 = 0.0139
Total = 0.111

Chunk B wins despite never being ranked highly anywhere — it was consistently relevant across all query variants. This is the insight: consistency beats one-off high ranks.

RRF is the standard for good reason — it requires no tuning and performs within 1-2% of learned fusion methods.

PART 6 — CROSS-ENCODER RE-RANKING
The fundamental accuracy gap
The bi-encoder (what you used for dense search) has a clever but flawed architecture:
Query  →  Encoder  →  Query vector  ─┐
                                      ├→ cosine similarity
Chunk  →  Encoder  →  Chunk vector  ─┘
Query and chunk are encoded separately. The model never sees them together. It can't notice that the chunk mentions the query topic but in a completely different context. It can't understand negation ("this is NOT about X"), qualification ("rarely related to X"), or complex interactions.
The cross-encoder sees them together:
[CLS] Query [SEP] Chunk [SEP]  →  Encoder  →  Single relevance score
The full attention mechanism runs over both simultaneously. Every word in the query can attend to every word in the chunk. It catches:

The chunk contains query terms but in an unrelated context
The chunk is about the query topic but contradicts rather than answers
Subtle qualifications ("in some cases, X — but for your use case, Y")


Why not use cross-encoder for everything?
Speed.
For dense search, you pre-compute chunk embeddings offline. At query time, you only embed the query (one forward pass) and do a vector similarity search (fast math). Total: ~10ms.
For cross-encoder, you must run a forward pass for every query-chunk pair at query time. You can't pre-compute anything. At 10,000 stored chunks, that's 10,000 forward passes per query. Total: minutes.
The solution your architecture uses: bi-encoder to get to top-20, cross-encoder only on those 20. This is called a retrieval-reranking pipeline or two-stage retrieval. Stage 1 (recall): get the probably-relevant candidates fast. Stage 2 (precision): precisely rank those candidates accurately.

The two re-ranking options
Local: cross-encoder/ms-marco-MiniLM-L-6-v2

Trained on MS MARCO passage ranking dataset (massive Microsoft QA dataset)
Runs on CPU, ~50-100ms for 20 chunks
Free, no API key, works offline
Good quality for general-purpose QA

Cohere Rerank API

API call, ~200ms latency
Costs money (~$1 per 1000 queries)
Slightly better quality for non-English, multilingual
Easier to implement (one API call vs local model management)

For a portfolio project: start with the local model. It keeps the system self-contained and shows you understand the technology. Add Cohere as an option flag.
Other alternatives:

ms-marco-electra-base — larger, more accurate, slower
bge-reranker-large — excellent, especially for Chinese/multilingual
Cohere Rerank v3 — current best API option
Jina Reranker — good API alternative to Cohere
GPT-4 as reranker — prompt it to score relevance 0-10, slow and expensive but highest quality


PART 7 — EVALUATION METRICS
Why you need an eval set
Without measurement, you're guessing. "The answers seem better" is not publishable, not credible, and not useful for understanding where your system fails. The eval set is what turns this from a demo into a serious project.

NDCG@5 — Normalized Discounted Cumulative Gain
The most standard metric in information retrieval.
DCG (raw version):
DCG@5 = Σ relevance(rank_i) / log2(rank_i + 1)
If you retrieve 5 chunks and the relevance scores (0=irrelevant, 1=relevant) are [1, 0, 1, 0, 1]:
DCG@5 = 1/log2(2) + 0/log2(3) + 1/log2(4) + 0/log2(5) + 1/log2(6)
      = 1.0 + 0 + 0.5 + 0 + 0.387 = 1.887
The denominator log2(rank+1) is the discount — a relevant doc at rank 1 is worth more than at rank 5. Getting the right answer at the top matters.
NDCG = DCG / IDCG, where IDCG is the DCG of the ideal ranking (all relevant docs first). This normalizes to 0-1.
For your project:

Naive RAG: NDCG@5 ≈ 0.45-0.55 (typical for basic dense retrieval)
Hybrid + reranking: NDCG@5 ≈ 0.65-0.75 (the improvement you'll measure)


MRR — Mean Reciprocal Rank
Simpler than NDCG. For each query: what rank was the first relevant document?
MRR = (1/|queries|) × Σ 1/rank_of_first_relevant_doc
If you have 3 queries and the first relevant doc appeared at ranks 1, 3, and 2:
MRR = (1/3) × (1/1 + 1/3 + 1/2) = (1/3) × 1.833 = 0.611
MRR answers: "How often does the right answer come first?" It's particularly useful when you only show the user one result or when rank 1 matters disproportionately.

Faithfulness (RAGAS)
NDCG and MRR measure retrieval quality — did you find the right chunks? Faithfulness measures generation quality — did the LLM actually use those chunks honestly?
RAGAS uses an LLM judge to answer: "For each claim in the generated answer, can that claim be directly inferred from the retrieved context?"
Faithfulness = (# claims supported by context) / (# total claims in answer)
This catches hallucination. A model might retrieve perfect chunks but then ignore them and make up an answer from its training data. Faithfulness = 0.6 means 40% of claims are hallucinated or unsupported.
Why naive RAG scores lower on faithfulness: With poor retrieval, the context is irrelevant. The LLM, lacking good context to ground itself, falls back on its parametric knowledge (training data). This looks like an answer but it's not grounded in your documents.
Other RAGAS metrics you can add:

Answer Relevancy — does the answer actually address the question?
Context Precision — are the retrieved chunks actually relevant to the question?
Context Recall — did you retrieve all the relevant information that exists?


Building the eval set
This is the hardest part. You have two options:
Manual: Write 50-100 queries yourself from your document corpus. For each, annotate which chunks are relevant. Time-consuming but highest quality.
LLM-generated: Prompt an LLM: "Given this chunk, generate 3 questions that this chunk would answer." Saves time but can be circular — the LLM generates questions the retriever finds easy.
Best practice: LLM generates candidates, human reviews and filters. The human filter matters — LLMs generate softball questions that don't test edge cases.

PART 8 — THE LLM LAYER (OpenRouter)
Why OpenRouter

There are 3 different things people mean by "eval"
1. Retrieval Eval (what this project does)
Did you fetch the right chunks from your document corpus?
You have ground truth: for query Q, chunk C is the correct answer. You measure whether your retrieval system actually returned chunk C, and at what rank.
Metrics: NDCG@5, MRR, Precision@K — all purely about did the right document come back?
This has nothing to do with the LLM. You could swap the LLM out entirely and the retrieval score wouldn't change.
2. Generation Eval (also what this project does, via RAGAS)
Given that you retrieved some chunks, did the LLM produce a good answer from them?
This is where Faithfulness lives — an LLM judge reads the generated answer and the source chunks and asks "did the model make up anything, or is every claim grounded in the context?"
Also includes Answer Relevancy — did the answer actually address the question, or did it go off on a tangent?
This is about LLM behavior on top of retrieved context.
3. Model Eval / LLM Eval (completely different world)
This is evaluating the LLM itself — benchmarks like MMLU, HumanEval, HellaSwag, GPQA. Testing whether a model can do math, write code, reason logically, pass medical exams.
This is what Anthropic, OpenAI, and Google do when they release a new model. It has nothing to do with RAG.

Layer 3 — The Data Layer (How Each RAG Stores Its Data)
Each RAG has three data structures that it owns.
The ChromaDB Collection
ChromaDB runs as a single local instance pointing at store/chroma_db/. Inside it, each RAG has its own named collection — medrag, legalrag, etc.
A collection stores, for each chunk:

The raw text of the chunk
Its 384-dimensional embedding vector
Metadata: source filename, page number, chunk index, chunk size

When you query, you pass in a query vector and ChromaDB returns the top-K most similar chunks by cosine similarity. That's the dense search side.
Collections are fully isolated. Querying medrag collection never touches legalrag data. Deleting a RAG means calling client.delete_collection("medrag") — clean and complete.
The BM25 Index
BM25 can't be stored inside ChromaDB — it's a completely separate data structure. The rank_bm25 library builds an in-memory index from a list of tokenized documents.
The problem: Python objects can't survive app restarts. So after building the index during ingestion, you serialize it with pickle and save it as store/medrag/bm25_index.pkl. On startup, when someone selects MedRAG, you deserialize it back into memory.
The index contains: the tokenized text of every chunk, their IDF weights, and their chunk IDs (which match the ChromaDB chunk IDs so you can merge results).
This chunk ID matching is crucial. When BM25 returns its ranked list and ChromaDB returns its ranked list, you need to merge them. Both systems must use the same IDs so RRF can reconcile "chunk_042 appeared at rank 3 in BM25 and rank 7 in dense search."
The Eval Set
store/medrag/eval_set.json contains the list of test queries:
[
  {
    query: "What is the first-line treatment for type 2 diabetes?",
    ideal_answer: "Metformin, unless contraindicated...",
    relevant_chunk_ids: ["chunk_042", "chunk_107"],
    source_doc: "diabetes_guidelines_2024.pdf"

Layer 4 — The Pipeline Object
This is the heart of the backend. When a RAG is selected, you instantiate a Pipeline object configured with that RAG's settings.
Pipeline
  └── knows its config (from config.json)
  └── holds a reference to the ChromaDB collection
  └── holds a reference to the BM25 index
  └── has methods:
       .ingest(file) → chunks → embeds → stores in both indexes
       .query(text) → returns answer + sources + internals
       .evaluate(eval_set) → returns metrics dict
The .query() method is where all the action happens. Here's the exact sequence of what happens when a user sends a message in the chat tab:
Step 1 — Query expansion (if enabled)
The raw query goes to the LLM. Returns 3 variants. You now have 4 queries.
Step 2 — Dense retrieval
Each of the 4 queries gets embedded. Each embedding goes into ChromaDB. You get 4 ranked lists of up to 20 chunks each, with cosine similarity scores.
Step 3 — Sparse retrieval (if BM25 enabled)
Each of the 4 queries gets tokenized. Each goes into the BM25 index. You get 4 more ranked lists.
Step 4 — RRF fusion
All 8 lists (or 4 if dense-only) get merged via RRF. Output: one ranked list of unique chunks with RRF scores.
Step 5 — Re-ranking (if enabled)
Top-20 chunks from RRF go into the cross-encoder as query-chunk pairs. Each pair gets a relevance score. Re-sort. Take top-5.
Step 6 — Context assembly
Top-5 chunks are formatted into a context block with source metadata.
Step 7 — LLM generation
Context + original query goes to OpenRouter. Response comes back with citations.
Step 8 — Return the full internals
The pipeline returns not just the answer, but everything that happened: what the query variants were, what each retrieval method returned, pre- and post-rerank rankings, scores at each stage. This is what the Internals tab displays.

Layer 5 — Session State (How Streamlit Holds It All Together)
Streamlit reruns the entire script on every user interaction. This means every variable is reset unless you explicitly persist it in st.session_state. Session state is a dictionary that survives reruns within one browser session.
Your session state has this shape:
st.session_state = {
  "active_rag_id": "medrag",
  "active_pipeline": <Pipeline object>,
  "active_tab": "chat",
  "chat_history": [
    { role: "user", content: "..." },
    { role: "assistant", content: "...", sources: [...] }
  ],
  "last_internals": { ... },
  "all_rags": [ ... ]   ← loaded from rags.json at startup
}
The active_pipeline object is the expensive thing. When a user clicks a different RAG in the sidebar, you check if the pipeline is already loaded for that RAG. If not, you instantiate a new Pipeline — which means loading the ChromaDB collection reference and deserializing the BM25 pickle from disk. This takes 1-3 seconds. After that, switching back to an already-loaded RAG is instant because the Pipeline object lives in session state.

Layer 6 — The Evaluator
The evaluator is a separate component that the pipeline calls. It does two things.
Retrieval metrics (NDCG, MRR) — these are computed entirely locally, no API calls needed. You run the retrieval stages of the pipeline (no LLM generation) for each eval query, get the ranked chunk list, compare it to the relevant_chunk_ids in the eval set, and compute the math. You also run naive retrieval (dense only, no expansion, no reranking) on the same queries to get the baseline.
Generation metrics (Faithfulness via RAGAS) — these require an LLM judge. RAGAS sends each generated answer along with its source chunks to an LLM and asks it to verify every claim. This is expensive — it makes one LLM call per eval query. On a 50-query eval set, that's 50 API calls just for the faithfulness score. This is why you cache results and don't run eval on every chat message.
The evaluator outputs one dictionary:
{
  naive: { ndcg: 0.56, mrr: 0.63, faithfulness: 0.60, avg_latency: 0.7 },
  hybrid: { ndcg: 0.71, mrr: 0.83, faithfulness: 0.79, avg_latency: 1.8 },
  per_query: [
    { query: "...", naive_rank: 3, hybrid_rank: 1, faithful: true },
    ...
  ]
}
This gets written to eval_results.json and the eval tab renders it.

The Config Lock Problem in Detail
When a user creates a RAG and then ingests documents, the embedding model is "locked in" because the vectors stored in ChromaDB were computed with that specific model. Model A's vector space is completely incompatible with Model B's. A query embedded with Model B will not meaningfully match chunks embedded with Model A.
Everything else in the config is actually modifiable without re-ingestion:

Chunking strategy — locked, because it determined how many chunks exist and where boundaries are
Embedding model — locked, vectors are model-specific
Chunk size / overlap — locked, same reason as chunking
Vector store type — locked, data lives in that store
BM25 — can be toggled on/off, can be rebuilt from raw chunk text stored in ChromaDB metadata
Fusion method — just math, change anytime
Query expansion — just an LLM call, change anytime
Re-ranker — just a model inference step, change anytime
Top-K values — change anytime
LLM — change anytime, it's the last step

So the locked components are exactly the ingestion-time decisions. Everything after the vector store is runtime configuration and can be modified live. This is actually a nice UX distinction — in the Config tab you could show two sections: "Ingestion config (locked)" and "Retrieval config (editable)."

At any given time, only one RAG's pipeline is active in the session. But all RAGs' data sits on disk simultaneously. The ChromaDB instance has multiple collections. The store/ folder has multiple RAG subfolders.

The only shared thing is the ChromaDB client object itself, which just points at the store/chroma_db/ directory. All the isolation is handled at the collection name level.

The Ephemeral Reality
When the container is destroyed and relaunched:

All RAGs gone — rags.json is fresh
All ChromaDB collections gone
All BM25 indexes gone
All eval sets and results gone
All chat history gone — SQLite file is fresh
All ingested documents gone

The only things that survive are the baked-in models and your code.
This is completely fine for a portfolio project. If you ever wanted persistence across restarts, you'd mount a Docker volume:
docker run -v /your/server/data:/app/store ...
That single flag makes everything persistent. But you said fresh on restart, so no volume mount — and that's a perfectly valid deployment choice.

Resource Requirements on the Server
This matters because you're running embedding models and a cross-encoder locally on CPU.
Minimum: 4 CPU cores, 8GB RAM
Comfortable: 8 CPU cores, 16GB RAM
The bottleneck is the cross-encoder re-ranker at query time. It runs 20 forward passes sequentially on CPU, which takes roughly 1-3 seconds depending on the server. Embedding a query is fast (~100ms). BM25 is instant. The LLM call to OpenRouter is the longest wait (~1-2s) but that's network, not your server.
During ingestion, embedding 1000 chunks on CPU with MiniLM takes roughly 30-60 seconds. Not fast, but ingestion is a one-time operation per document.
RAM breakdown:

MiniLM model in memory: ~500MB
Cross-encoder model in memory: ~400MB
ChromaDB with 10k chunks: ~500MB
BM25 index for 10k chunks: ~200MB
Streamlit + Python overhead: ~500MB
Total comfortable ceiling: ~3-4GB RAM

So an 8GB server is genuinely comfortable. A 4GB server works but leaves little headroom.

One Thing to Decide Now
Do the models load at container startup or on first use?
Option A — Load all models at startup:

Container takes 20-30 seconds to become ready
Once ready, first query is instant
All model RAM is consumed whether you use them or not

Option B — Load models lazily on first use:

Container starts instantly
First query after startup has a 5-10 second delay while model loads into RAM
Only the models you actually select get loaded

For a portfolio demo where you're showing it to people, Option A is better — you never want a 10-second lag mid-demo. For a server running 24/7 with infrequent use, Option B saves RAM.
The clean way to implement this is a model cache singleton — a module-level dictionary that maps model names to loaded model objects. First call loads and caches. Every subsequent call returns the cached object instantly. Startup pre-warms this cache for the default models.

--- core/chunker.py ---

  chunk_document(text: str, strategy: str, chunk_size: int, overlap: int) 
    → List[str]

  Implement all four strategies:

  "fixed": Split by character count with overlap. 
    Use langchain_text_splitters.CharacterTextSplitter if available,
    else implement manually.

  "recursive": Split preferring \n\n, then \n, then ". ", then " ".
    Stop when chunk is under chunk_size tokens.
    Implement without langchain dependency — write the recursive logic manually.

  "sentence": Use basic sentence splitting on ". ", "! ", "? ".
    Group sentences until token count approaches chunk_size.
    Token count approximation: len(text.split()) * 1.3

  "semantic": Embed every sentence. Find split points where cosine 
    similarity between adjacent sentence embeddings drops below 0.5.
    Group resulting segments to stay under chunk_size.
    This is expensive — only offered for small documents.

  All strategies must return List[str] with no empty strings.

class Pipeline:
  __init__(self, config: RAGConfig, api_key: str)
    self.config = config
    self.collection = get_or_create_collection(config.id, config.embedding_model)
    self.bm25_index, self.bm25_chunk_ids = load_bm25_index(config.id) or (None, [])

  ingest_text(self, text: str, source_name: str, page_number: int | None) → int
    Returns number of chunks created.
    - chunk text using config chunking settings
    - generate chunk_ids: f"{self.config.id}_chunk_{global_counter}"
    - embed all chunks in batches of 64
    - add to ChromaDB
    - rebuild BM25 index from scratch (get all texts from ChromaDB, add new ones)
    - save BM25 index
    - update config doc_count and chunk_count in registry

  query(self, user_query: str) → dict
    Returns full internals dict:
    {
      answer: str,
      sources: List[dict],
      query_variants: List[str],
      pre_rerank_results: List[{chunk_id, text, source, rrf_score, dense_score, bm25_rank}],
      post_rerank_results: List[{chunk_id, text, source, cross_encoder_score, rank_before, rank_after}],
      latency_breakdown: {expansion_ms, retrieval_ms, rerank_ms, generation_ms}
    }

    Step by step:
    1. t0 = time. Expand query per config. t1 = time. expansion_ms = t1-t0
    2. t0. Hybrid retrieve top_k_retrieval chunks. Get chunk texts from ChromaDB. t1. retrieval_ms.
    3. If reranker != "none": t0. Rerank. t1. rerank_ms.
    4. Take top_k_final chunks.
    5. t0. Generate answer. t1. generation_ms.
    6. Assemble and return full internals dict.

  rebuild_bm25(self) → None
    Fetch all chunks from ChromaDB. Rebuild index. Save pickle.

  get_all_chunk_texts(self) → List[Tuple[str, str]]
    Returns list of (chunk_id, text) for all chunks in this RAG's collection.

    For each query in eval_set:
      - Run naive_retrieve → get ranked chunk_ids → compute NDCG@5, MRR
      - Run pipeline.query() → get ranked results → compute NDCG@5, MRR
      - Score faithfulness via RAGAS (or simple LLM judge if RAGAS fails)
      - Record latency for both
    Aggregate. Return eval_results dict.
    Save to store/{rag_id}/eval_results.json.

  compute_ndcg_at_5(
    retrieved_ids: List[str],
    relevant_ids: List[str]
  ) → float

  compute_mrr(
    retrieved_ids: List[str],
    relevant_ids: List[str]
  ) → float

  Note on RAGAS: RAGAS may require specific setup. If import fails, 
  fall back to a simple LLM judge prompt that scores faithfulness 0-1:
  "Given this answer and these source passages, what fraction of claims 
  in the answer are supported by the passages? Return only a float 0-1."

render_ingest(pipeline: Pipeline, config: RAGConfig):
  
  st.subheader("Add Documents")
  
  Tab1, Tab2, Tab3 = st.tabs(["Upload File", "Paste URL", "Paste Text"])
  
  with Tab1:
    uploaded = st.file_uploader("Upload PDF, TXT, or DOCX", 
      type=["pdf", "txt", "docx"])
    if uploaded and st.button("Ingest File"):
      with st.spinner("Chunking and embedding..."):
        text = extract_text_from_file(uploaded)
        n_chunks = pipeline.ingest_text(text, uploaded.name, None)
      st.success(f"Ingested {n_chunks} chunks from {uploaded.name}")
      st.rerun()
  
  with Tab2:
    url = st.text_input("URL")
    if url and st.button("Scrape and Ingest"):
      with st.spinner("Fetching and embedding..."):
        text = scrape_url(url)
        n_chunks = pipeline.ingest_text(text, url, None)
      st.success(f"Ingested {n_chunks} chunks from {url}")
      st.rerun()
  
  with Tab3:
    pasted = st.text_area("Paste plain text", height=200)
    doc_name = st.text_input("Document name")
    if st.button("Ingest Text"):
      n_chunks = pipeline.ingest_text(pasted, doc_name or "pasted_text", None)
      st.success(f"Ingested {n_chunks} chunks")
      st.rerun()
  
  st.divider()
  st.subheader("Ingested Documents")
  
  Fetch all chunk metadata from ChromaDB.
  Group by source_doc.
  For each document show: name, chunk count, ingest time.
  Use st.dataframe for the table.
  
  st.caption(f"Total: {config.chunk_count} chunks across {config.doc_count} documents")

  Helper functions in this file:
  extract_text_from_file(uploaded_file) → str
    PDF: use pypdf PdfReader
    TXT: decode bytes
    DOCX: use python-docx
  
  scrape_url(url: str) → str
    requests.get(url, timeout=10)
    BeautifulSoup(response.text, "html.parser")
    Extract text from p, h1-h6, li tags.
    Return joined text.

render_chat(pipeline: Pipeline, config: RAGConfig, api_key: str):
  
  If config.chunk_count == 0:
    st.warning("Ingest documents first before chatting.")
    return

    For user messages: st.chat_message("user")
    For assistant messages: st.chat_message("assistant")
      Show answer text.
      Show source chips below answer: small colored labels showing source_doc + page.
  
  user_input = st.chat_input("Ask a question about your documents...")
  
  If user_input:
    Save user message to DB.
    Display immediately with st.chat_message("user").
    
    with st.chat_message("assistant"):

          )

    Save assistant message + sources to DB.

render_internals(config: RAGConfig):
  
  internals = st.session_state.get("last_internals")
  
  If not internals:
    st.info("Send a message in the Chat tab to see retrieval internals here.")
    return
  
  st.subheader("Last query internals")
  st.code(internals["query_variants"], language=None)
  
  col1, col2 = st.columns(2)
  
  with col1:
    st.caption("Before re-ranking (RRF top results)")
    for i, chunk in enumerate(internals["pre_rerank_results"]):
      with st.expander(f"#{i+1} — {chunk['source']} (RRF: {chunk['rrf_score']:.4f})"):
        st.markdown(chunk["text"][:500] + "..." if len(chunk["text"]) > 500 else chunk["text"])
        st.caption(f"Dense score: {chunk['dense_score']:.3f} | BM25 rank: {chunk['bm25_rank']}")
  
  with col2:
    st.caption("After re-ranking")
    for i, chunk in enumerate(internals["post_rerank_results"]):
      delta = chunk["rank_before"] - (i + 1)
      delta_str = f"↑{delta}" if delta > 0 else f"↓{abs(delta)}" if delta < 0 else "="
      with st.expander(f"#{i+1} — {chunk['source']} ({delta_str}) score: {chunk['cross_encoder_score']:.3f}"):
        st.markdown(chunk["text"][:500] + "..." if len(chunk["text"]) > 500 else chunk["text"])
  
  st.divider()
  st.subheader("Latency breakdown")
  cols = st.columns(4)
  cols[0].metric("Query expansion", f"{internals['latency_breakdown']['expansion_ms']}ms")
  cols[1].metric("Retrieval", f"{internals['latency_breakdown']['retrieval_ms']}ms")
  cols[2].metric("Re-ranking", f"{internals['latency_breakdown']['rerank_ms']}ms")
  cols[3].metric("Generation", f"{internals['latency_breakdown']['generation_ms']}ms")

- On failure: return a graceful error dict, never raise to Streamlit
- Log the error with loguru

Every file read must handle FileNotFoundError.

ChromaDB errors: if collection operation fails, log and surface as

BM25 search with no index: return empty list, log warning.

Query expansion failure: fall back to [original_query] silently, log warning.
